init_smart_speaker.sql 表设计总结

一、总体设计思路

这份 `init_smart_speaker.sql` 设计的是一个“智能音箱系统”的关系型数据库，核心目标是支持以下几类业务：

1. 用户管理
2. 设备管理
3. 用户与设备绑定
4. 音乐映射与播放历史
5. 用户反馈与偏好行为
6. 每日统计汇总

从数据库建模角度看，这份设计采用的是“基础主数据表 + 业务行为表 + 统计汇总表”的结构：

- 基础主数据表：`user`、`device`、`action_dict`
- 关系表：`user_device_binding`、`friendship`
- 业务行为表：`auth_token`、`media_mapping`、`play_history`、`user_feedback`
- 汇总统计表：`Daily_Stats`

这样设计的原理是把“实体”“关系”“行为”“统计结果”分层管理，避免把所有信息混在一张大表里，提高可维护性、扩展性和查询效率。


二、表是怎么设计的

1. `user`

用途：
- 保存系统用户基本信息。

字段设计：
- `user_id`：用户主键，自增
- `username`：用户名
- `password_hash`：密码哈希值
- `phone`：手机号，可空
- `created_at`：创建时间

设计原理：
- 用户是整个系统的核心实体之一，因此使用独立主表。
- 主键采用自增 `BIGINT`，便于后续扩容。
- 密码不直接存明文，而是存 `password_hash`，符合安全设计原则。
- `created_at` 自动记录创建时间，方便审计和排序。

为什么这样设计：
- 用户信息会被很多表引用，例如播放历史、反馈、绑定设备、令牌等，所以必须独立成主表。
- 使用哈希密码是最基本的安全要求。


2. `device`

用途：
- 保存智能音箱设备信息。

字段设计：
- `device_id`：设备主键，自增
- `device_number`：设备编号
- `model_name`：设备型号
- `status`：设备状态
- `last_active`：最后活跃时间

设计原理：
- 设备是独立业务实体，和用户不是一对一死绑定关系，因此需要单独建表。
- `last_active` 用于表示设备近期是否在线、最近一次交互时间。

为什么这样设计：
- 后续一个设备可能被多个用户使用，也可能频繁切换主人，所以设备信息必须单独管理。


3. `user_device_binding`

用途：
- 保存用户和设备之间的绑定关系。

字段设计：
- `user_id`
- `device_id`
- `custom_device_name`
- `is_primary`
- 联合主键：`(user_id, device_id)`

设计原理：
- 这是典型的多对多中间表。
- 一个用户可以绑定多个设备。
- 一个设备也可能与多个用户有关联。
- 使用联合主键，保证同一个用户和同一个设备的绑定关系不会重复。

为什么这样设计：
- 中间表模式是解决多对多关系最标准的方法。
- `custom_device_name` 用于保存用户自定义名称。
- `is_primary` 用于标识主设备，便于业务侧快速判断默认设备。
- `custom_device_name` 使用 `utf8mb4_bin`，说明这里更偏向严格区分大小写的名字存储。


4. `friendship`

用途：
- 保存用户之间的好友关系。

字段设计：
- `user_id_1`
- `user_id_2`
- 联合主键：`(user_id_2, user_id_1)`

设计原理：
- 这是用户到用户的自关联关系表。
- 两列都引用 `user` 表。

为什么这样设计：
- 朋友关系本质上不是用户表里的一个字段，而是一种“用户和用户之间的关系”。
- 用关系表表示用户社交关系，比在用户表里硬塞好友字段更合理。


5. `auth_token`

用途：
- 保存用户接入外部平台的授权令牌信息。

字段设计：
- `auth_id`：主键，自增
- `user_id`：关联用户
- `platform_type`：平台类型
- `access_token`
- `refresh_token`
- `expires_at`

设计原理：
- 授权信息属于用户的外部平台接入凭证，不适合直接放在用户表中。
- 一个用户未来可能接多个平台，因此拆成独立表更有扩展性。

为什么这样设计：
- 令牌数据长度不固定，而且更新频率和用户基本信息不同。
- 单独建表能减少用户主表膨胀。


6. `media_mapping`

用途：
- 保存系统内歌曲与外部平台歌曲之间的映射信息。

字段设计：
- `mapping_id`：主键，自增
- `user_id`：关联用户
- `song_title`
- `artist`
- `platform`
- `external_id`
- `cover_url`

设计原理：
- 音乐播放通常依赖第三方平台，因此需要保存本系统与外部音乐资源的映射关系。
- `external_id` 用来对接外部平台歌曲 ID。

为什么这样设计：
- 歌曲信息和用户信息不是一类数据。
- 后续一首歌可能来自不同平台，所以需要 `platform + external_id` 这类组合信息。
- 单独的映射表便于缓存和查询，避免每次播放时都去实时请求外部平台。


7. `play_history`

用途：
- 保存用户实际播放过的歌曲历史记录。

字段设计：
- `history_id`：主键，自增
- `device_id`：关联设备
- `user_id`：关联用户
- `mapping_id`：关联歌曲映射
- `play_duration`：播放时长
- `created_at`：播放时间
- `style`：来源或类型

设计原理：
- 播放历史属于典型的“行为明细表”。
- 每一次播放都是一条独立记录，因此用自增主键最合理。
- 表中同时关联用户、设备、歌曲，构成完整播放事件。

为什么这样设计：
- 如果不用独立的 `history_id`，而用用户+设备做主键，会导致同一用户同一设备无法保存多条播放记录。
- 使用明细表设计，才能支撑后续统计、推荐、排行、每日汇总等功能。

特别说明：
- `mapping_id` 外键采用 `ON DELETE SET NULL`
- 原理是：如果某首歌曲映射被删除，历史记录本身不应该被删，只是歌曲引用失效
- 这是“保留历史、允许关联对象消失”的典型设计


8. `user_feedback`

用途：
- 保存用户反馈内容。

字段设计：
- `feedback_id`：主键，自增
- `user_id`：关联用户
- `content`：反馈内容

设计原理：
- 反馈属于用户行为记录，不应放在用户主表中。
- 一名用户可以有多条反馈，因此拆成单独表。

为什么这样设计：
- 用户反馈通常是追加式数据，和用户主数据更新逻辑不同。
- 独立表更适合后续统计意见、人工审核或问题追踪。


9. `action_dict`

用途：
- 保存动作字典。

字段设计：
- `action_id`
- `action_code`
- `action_name`
- `category`

设计原理：
- 这是字典表，用于保存系统内部动作类型的标准定义。

为什么这样设计：
- 把动作枚举独立成字典表，便于统一维护，不需要在代码和数据库中到处写死字符串。
- 后续如果扩展新的动作分类，也更容易管理。


10. `Daily_Stats`

用途：
- 保存每天的播放统计汇总结果。

字段设计：
- `stat_date`：主键，统计日期
- `total_play_count`
- `unique_song_count`
- `unique_user_count`
- `unique_device_count`
- `total_play_duration_seconds`
- `avg_play_duration_seconds`
- `hottest_song_external_id`
- `hottest_song_name`
- `hottest_artist`
- `hottest_play_count`
- `generated_at`
- `updated_at`

设计原理：
- 这是典型的汇总表，不是明细表。
- 一天只保留一条统计结果，所以主键用 `stat_date`。
- 汇总表的数据主要从 `play_history` 聚合而来，再结合 `media_mapping` 补充热门歌曲信息。

为什么这样设计：
- 如果每次展示日报都实时扫 `play_history`，随着数据增长，查询会越来越慢。
- 预先生成 `Daily_Stats` 能显著提升统计类接口和页面的性能。
- `generated_at` 和 `updated_at` 用于记录统计结果何时生成、何时更新，便于运维和排错。


三、表与表之间的关系是怎么设计的

1. `user` 是核心主表

与它有关系的表包括：
- `auth_token`
- `friendship`
- `media_mapping`
- `play_history`
- `user_device_binding`
- `user_feedback`

设计原理：
- 用户是系统操作主体，因此很多业务表都需要引用用户。


2. `device` 是设备主表

与它有关系的表包括：
- `play_history`
- `user_device_binding`

设计原理：
- 设备是行为发生载体，所以播放记录和绑定关系都需要依赖它。


3. `media_mapping` 是歌曲映射主表

与它有关系的表包括：
- `play_history`
- `Daily_Stats`（逻辑上依赖，不是物理外键）

设计原理：
- 播放记录引用具体歌曲映射；
- 每日统计需要热门歌曲信息。


4. `Daily_Stats` 是派生表，不是普通子表

重要点：
- 它不应该像普通业务表那样和 `play_history` 建物理 FK。
- 它是从 `play_history` 聚合计算出来的。

设计原理：
- 明细表和汇总表的关系是“派生关系”，不是“直接引用关系”。
- 所以在 ER 图中更适合用逻辑连线或 Link，而不是 Reference 外键。


四、这份设计背后的总体原理

1. 实体和行为分离

用户、设备、歌曲映射是相对稳定的“实体数据”；  
播放历史、反馈、授权令牌是不断新增或变化的“行为数据”。

这样分离的好处：
- 表职责清晰
- 查询更有针对性
- 后期维护更容易


2. 多对多关系通过中间表解决

例如：
- 用户和设备之间不是简单一对一
- 所以用 `user_device_binding` 解决

这是关系型数据库里最标准的设计方式。


3. 历史数据使用明细表 + 自增主键

例如：
- `play_history`
- `user_feedback`

这样做的原理是：
- 历史行为是可以无限增长的
- 每条记录都应独立存在
- 自增主键最适合这种场景


4. 汇总数据单独落表

例如：
- `Daily_Stats`

原理是：
- 用空间换时间
- 提前算好统计结果，减少在线查询开销


5. 数据一致性由数据库约束保证

脚本中使用了：
- 主键约束
- 外键约束
- 非空约束
- 默认值

这样做的原因是：
- 把数据正确性尽量下沉到数据库层
- 减少程序侧漏校验造成的脏数据


五、为什么这份设计是合理的

这份设计合理，主要体现在以下几点：

1. 能覆盖智能音箱项目的主要业务链路
- 用户
- 设备
- 歌曲
- 播放
- 反馈
- 统计

2. 表职责划分比较清楚
- 主表负责保存核心对象
- 关系表负责描述关联
- 明细表负责记录行为
- 汇总表负责加速统计

3. 具备较好的扩展性
- 后续可增加平台类型
- 可增加更多统计表
- 可增加更多设备状态或动作字典

4. 适合后端接口开发
- 主键清晰
- 外键明确
- 自增字段便于插入
- 字段命名比较直观


六、总结

这份 `init_smart_speaker.sql` 的设计，本质上是在为一个“智能音箱音乐系统”建立一套规范的关系型数据模型。

它的设计重点不是只把表建出来，而是：

- 用主表保存稳定实体
- 用关系表描述用户和设备、用户和用户之间的关系
- 用行为表记录播放、反馈、授权等动态数据
- 用汇总表提升统计性能
- 用主键、外键、默认值和非空约束保证数据质量

因此，这份设计既符合关系型数据库规范，也比较符合实际业务开发逻辑，是一套“以用户、设备、播放行为和统计分析为中心”的数据库设计方案。
