## 6. 安全设计

后台接口使用独立的管理员 Token，不和小程序用户 Token 混用。

实现方式：

1. 登录成功后，后端生成带过期时间的 HMAC Token。
2. 前端访问后台接口时传：

```http
Authorization: Bearer <token>
```

3. 后端从 Token 中解析管理员身份和角色。
4. 每个后台接口按角色做权限判断：
   - 未登录返回 `401`
   - 跨角色访问返回 `403`
   - 有权限才返回业务数据

这样可以保证不同用户登录后只看到自己角色对应的后台功能。

### 7.4 Gunicorn 运行方式

`docker-compose.yml` 中已经配置：

```bash
GATEWAY_USE_GUNICORN=true
APP_HOST=0.0.0.0
APP_PORT=5000
GATEWAY_WORKERS=4
GATEWAY_THREADS=8
GATEWAY_TIMEOUT=30
APP_LISTEN_QUEUE=1024
```

容器启动后会执行：

```bash
python run_gateway.py
```

`run_gateway.py` 会在 Linux 服务器上自动使用 Gunicorn：

```bash
gunicorn app:app \
  --bind 0.0.0.0:5000 \
  --worker-class gthread \
  --workers 4 \
  --threads 8 \
  --backlog 1024 \
  --timeout 30
```

这符合 PPT 中“生产环境使用 Gunicorn，不直接使用 Flask app.run”的要求。

### 7.5 Nginx 反向代理

`nginx_api_musicplayer.conf` 的核心逻辑是：

```nginx
location / {
    proxy_pass http://127.0.0.1:5000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto https;
}
```

部署后访问流程是：

```text
浏览器/前端
  -> Nginx 443 HTTPS
  -> Gunicorn 127.0.0.1:5000
  -> Flask app.py
  -> admin_routes.py / 原有小程序接口 / 原有网易云相关服务
```

这样 Nginx 负责 HTTPS、反向代理和入口流量，Gunicorn 负责 Python Web 并发，Flask 只负责业务接口。

## 8. 高并发处理方式

本项目高并发处理不是只靠一个点，而是分层处理。

### 8.1 Nginx 入口层

Nginx 位于最外层：

- 处理 HTTPS。
- 反向代理到 Gunicorn。
- 可继续扩展为多个后端实例的负载均衡。
- 可以配置静态资源缓存、Gzip、请求大小限制、超时等。

### 8.2 Gunicorn Web 层

后端 Web 服务使用 Gunicorn：

- `workers=4`：多进程利用多核 CPU。
- `threads=8`：每个进程多个线程，提高并发请求处理能力。
- `worker-class=gthread`：适合当前 Flask 同步接口和数据库 I/O 场景。
- `backlog=1024`：提高连接排队能力。
- `timeout=30`：避免慢请求长期占用 worker。

按当前配置，理论并发处理单元约为：

```text
4 workers * 8 threads = 32 个并发执行线程
```

后续如果服务器 CPU 和内存更强，可以调整：

```bash
GATEWAY_WORKERS=CPU核心数 * 2 + 1
GATEWAY_THREADS=4 到 8
```

### 8.3 后台统计接口缓存

这次新增的后台统计接口中，对用户数、设备数、热歌、反馈、设备列表等查询增加了短 TTL 缓存：

```bash
ADMIN_CACHE_TTL_SECONDS=30
```

这样前端仪表盘频繁刷新时，不会每一次请求都直接打 MySQL。

适用场景：

- 数据总览
- 趋势分析
- 用户画像
- 区域热力图
- 热歌排行
- 反馈列表
- 设备列表

这些数据不要求毫秒级实时，30 秒缓存可以明显降低数据库压力。

### 8.4 消息队列削峰

原项目已有 RabbitMQ 和 `BufferedMessageDispatcher`：

- 设备上报接口先进入内存队列。
- 后台线程再投递到 RabbitMQ。
- Worker 异步消费并写入 MySQL/MongoDB。
- 当队列满时返回 `503`，提示前端或设备稍后重试。

这符合 PPT 中“异步处理、消息队列、避免阻塞主线程”的高并发思路。

### 8.5 Worker 水平扩展

`docker-compose.yml` 中已有多个 worker：

- `worker-writer`
- `worker-logger`
- `worker-validator`
- `worker-reader`
- `auto-scaler`

`auto-scaler.py` 会根据 RabbitMQ 堆积量和 gateway 指标决定扩缩容 worker，避免高峰期写入积压。

### 8.6 监控指标

原项目已有：

```http
GET /internal/metrics
```

指标包括：

- `accepted_per_second`
- `accepted_in_window`
- `dispatcher_queue_size`
- `process_count`

多进程 Gunicorn 下，项目通过 `GATEWAY_METRICS_MULTIPROCESS=true` 和多个 metrics 文件聚合统计。

### 8.7 数据库压力控制

后台接口优先聚合查询和兜底数据：

- 能查 MySQL 就返回真实数据。
- MySQL 暂时不可用时返回稳定演示数据，避免前端页面空白。
- 查询结果短时间缓存，降低重复读取。
- 设备上报走 RabbitMQ 异步写入，不和后台查询互相阻塞。
